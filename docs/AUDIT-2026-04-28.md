## 3. Schema↔UI invariants

Tools: two-sided contract tests.

### Backend side — `backend/tests/test_contracts/test_aggregator_contracts.py`

A reusable shape-walker (`assert_shape`) lets us declare each aggregator's expected response as a Python literal that mirrors the frontend's TypeScript interface. Walker primitives:
- `REQUIRED` — field must be present (any value, including null)
- `REQUIRED_NONNULL` — field must be present AND non-null
- `OPTIONAL` — field may be absent
- `list_of(spec)` — field is a list whose every element matches the inner spec
- nested dict — recursive

Aggregators covered (7):
- `GET /api/v1/today/summary`
- `GET /api/v1/path/summary`
- `GET /api/v1/promotion/summary`
- `GET /api/v1/catalog/`
- `GET /api/v1/exercises`
- `GET /api/v1/chat/notebook`
- `GET /api/v1/srs/due`

Each test registers a brand-new user (no fixture pollution), hits the endpoint, and walks the response. Six self-tests guard the walker itself (catches missing required, catches null in non-null, allows optional missing, walks into lists, aggregates multiple errors, handles nested objects).

**Result: 13/13 tests pass on a fresh user.** Drift in either direction now fails CI.

### Frontend side — `frontend/src/test/contracts/api-shape.test.ts`

A `shape()` helper walks a TypeScript fixture and emits its sorted-key, primitive-type-tagged shape. Each public response interface gets a fixture and a Vitest snapshot. If a backend field is added/renamed/removed and the matching TS interface drifts, the snapshot fails until intentionally regenerated.

Interfaces locked (7):
- `TodaySummaryResponse`
- `PathSummaryResponse`
- `PromotionSummaryResponse`
- `CatalogResponse`
- `ExerciseResponse`
- `NotebookEntryOut`
- `SRSCard`

**Result: 7/7 snapshots written, stable across re-runs.**

The two suites are complementary: backend test catches "server stopped sending field X"; frontend test catches "interface X added field Y but server doesn't know about it." Together they prevent silent drift in either direction.

**Sync rule (also in `AGENT-OPERATING-SPEC.md`):** when an aggregator interface changes, both the matching backend spec AND the matching frontend fixture/snapshot MUST change in the same PR.

## 4. Markdown / preview leaks

Audited every place a screen displays a truncated user/assistant content as a small label or preview tile.

### Findings

| Surface | Source | Status |
|---|---|---|
| Notebook card preview | `notebook-screen.tsx:42` | ✅ Already fixed in commit `3e8e988` (PR0/Notebook2) — uses `stripMarkdownToText` |
| Notebook detail "your note" | `note-detail-modal.tsx` | ✅ Already fixed in commit `3e8e988` |
| **Chat sidebar conversation title** | `chat/page.tsx:3093` (`onFirstMessage`) | 🚨 **Was leaking — fixed this PR** |
| Tutor screen "Recent conversations" rail | reads `c.title` | ✅ Same backend title — fixed transitively by the chat sidebar fix |
| Today micro-wins | `w.label` | ✅ Backend-controlled labels (cohort events), no user markdown |
| Today capstone preview | `capstone.title` | ✅ Backend-controlled title |
| Suggested-prompt cards | static text | ✅ Static |
| Chat conversation rename input | direct user typing | ✅ User explicitly chose the title |

### Fix

`chat/page.tsx` line 3093 was synthesizing the conversation title from `last.content.slice(0, 60)` — raw markdown straight into the conversation list. Now passes through the same `stripMarkdownToText → truncateAtWord` pipeline that powers the Notebook preview cards. The same fix transparently cleans the v8 Tutor screen "Recent conversations" rail because both surfaces read the same backend `conv.title`.

3 regression tests added in `markdown-text.test.ts` so the fix can't be reverted by accident.
